# MOD Dependency Workflow Design (Folder-Based PWAD Entry)

Status: proposed  
Last updated: 2026-03-07

## Purpose
- External PWADs are not bundled.
- Users install third-party content manually.
- Runtime management is based on `1 folder = 1 mod entry`.
- Active mod policy remains `single active entry only`.
- Users are not expected to edit JSON or any config file manually.

## Core Decision
- `wad/mods` is no longer a flat list of playable `.wad` files.
- Each direct child folder under `wad/mods` represents one candidate mod entry.
- A playable entry may launch one or more `.wad` files.
- Root-level files in `wad/mods` are reserved for guide text such as `00_HOW_TO_ADD_MODS_*.txt`.
- Internal per-entry config is generated by the game and stored under `BepInEx/config/chitsii.elin_justdoomit/profiles`, not under `wad`.

## User-Facing Folder Layout

```text
wad/
  freedoom1.wad
  freedoom2.wad
  mods/
    00_HOW_TO_ADD_MODS_EN.txt
    00_HOW_TO_ADD_MODS_JA.txt
    00_HOW_TO_ADD_MODS_ZH-CN.txt
    alien_vendetta/
      AV.WAD
      AVMOVFIX.WAD
    memento_mori/
      MM.WAD
      MMMUS.WAD
    no_end_in_sight/
      NEIS.wad
```

## Internal Storage

```text
BepInEx/config/chitsii.elin_justdoomit/profiles/
  runtime_loadout.json
  mod_rules.json
  mod_entry_configs/
    alien_vendetta.json
    memento_mori.json
```

- Users are not asked to open or edit these files.
- This avoids write failures on read-only installs.

## Non-Goals
- Multiple active mod entries at the same time
- In-game file manager
- Auto-download from mod sites
- Source-port-specific formats such as `.pk3` / `.pk7`
- Full expert-grade load-order editor in v1

## Discovery Model

### Candidate Scan
- Scan only direct child directories of `wad/mods`.
- Ignore files directly under `wad/mods` except guide text.
- Ignore hidden or disabled directories.

### Disabled / Reserved Folder Names
- Never show folders whose names:
  - start with `.`
  - start with `_`
  - equal `__MACOSX`

### Allowed Guide / Metadata Files
- For wrapper-folder rescue, guide/metadata files mean direct files with these extensions only:
  - `.txt`
  - `.md`
  - `.nfo`
  - `.diz`
  - `.jpg`
  - `.png`

### Content Root Resolution
- Each candidate folder resolves to exactly one content root.
- Resolution order:
  1. if the folder itself contains direct `.wad` files, use the folder itself
  2. otherwise, if the folder has exactly one child directory and all direct files in the parent use allowed guide/metadata extensions, and that child contains direct `.wad` files, use that child as the content root
  3. otherwise, resolution fails

### Entry States
- `ready-single`
  - exactly one direct `.wad` exists
- `setup-needed`
  - multiple direct `.wad` files exist and no valid internal config exists yet
- `ready-multi`
  - multiple direct `.wad` files exist and a valid internal config exists
- `error-config`
  - internal config exists but is invalid
- `error-unsupported`
  - the resolved content root contains unsupported non-WAD launch inputs such as `.deh`, `.bex`, `.pk3`, or `.pk7`
- `error-layout`
  - folder exists but layout is not recognized well enough to build a content root
- `hidden`
  - empty folder / no `.wad` / reserved folder

### Visibility Rules
- Show in the list:
  - `ready-single`
  - `ready-multi`
  - `setup-needed`
  - `error-config`
  - `error-unsupported`
  - `error-layout`
- Hide from the list:
  - `hidden`

Rationale:
- Users should see actionable setup failures.
- Pure junk folders should not pollute the list.

## Internal Entry Config

### Storage Path
- `BepInEx/config/chitsii.elin_justdoomit/profiles/mod_entry_configs/<entry_id>.json`

### Ownership
- Generated by the game
- Updated by the game
- Not documented as a file users should edit

### Purpose
- Store selected launch set for multi-WAD entries
- Store launch order
- Store optional display name override

### Minimal Shape

```json
{
  "display_name": "Alien Vendetta",
  "main_wad_file": "AV.WAD",
  "wad_order": ["AV.WAD", "AVMOVFIX.WAD"]
}
```

### Rules
- `main_wad_file` is required for multi-WAD entries
- `wad_order` is required for multi-WAD entries
- only `.wad` files listed in `wad_order` are launched
- unlisted `.wad` files are ignored for launch and classification
- `main_wad_file` must be included in `wad_order`
- listed files must exist directly under the content root
- duplicate names are invalid
- names are matched case-insensitively on Windows
- if `display_name` is missing or blank, use `entry_id`

### Config Identity
- Internal config is attached by `entry_id`.
- Changing the direct child folder name under `wad/mods` creates a different config identity.

Reason:
- This avoids lookup cycles and keeps the model simple.
- Resetting setup should be handled in-game, not by asking users to manage files.

## First-Time Multi-WAD Setup Flow

### Setup Trigger
- When a multi-WAD folder with no valid internal config is found, mark it as `[SETUP]`.

### Setup Dialog
- Selecting `[SETUP]` opens a setup dialog showing:
  - folder name
  - detected `.wad` files
  - proposed launch order

### Proposed Default
- Default launch order is alphabetical by filename.

### Minimum v1 User Controls
- `USE DETECTED ORDER`
- `MOVE UP`
- `MOVE DOWN`
- `EXCLUDE / INCLUDE`
- `BACK`

Reason:
- Users should not edit JSON.
- But they still need a way to fix wrong automatic order.

### Result
- On confirm, the game writes the internal config under `mod_entry_configs/<entry_id>.json`
- In the same transaction, the game must:
  - rebuild the entry
  - recompute `manifest_hash`
  - rerun dependency classification
  - rerender the list row
- The entry becomes `ready-multi` immediately without requiring the player to close and reopen `CONFIGURE MODS`

### Reset Setup Action
- `RESET SETUP` is not shown inside the first-time `[SETUP]` dialog.
- It is shown only for configured `ready-multi` entries inside `CONFIGURE MODS`.
- Placement rule:
  - render it as a separate action below the file list and below normal toggle/selection actions
  - keep it away from the main select/toggle hit area
  - it must not be easy to hit by mistake while navigating the ordinary mod list
- Selecting `RESET SETUP` opens a confirmation dialog.
- Minimum v1 confirmation text:
  - `Reset setup for this mod?`
  - `Selected files and order will be forgotten.`
- Minimum v1 confirmation choices:
  - `NO / BACK`
  - `YES / RESET`
- On confirmed reset, the game must:
  - delete `mod_entry_configs/<entry_id>.json`
  - clear current selection if that entry was selected
  - rebuild the entry immediately
  - rerender the row as `[SETUP]`
  - update the detail area immediately

Reason:
- Reset is a destructive action and should not live in the same hit area as normal mod selection.
- A confirmation step is required to prevent accidental loss of setup.

### Main PWAD Rule
- On first setup confirmation, set `main_wad_file` to the first included WAD in the confirmed order.
- Later reorder operations do not change `main_wad_file` automatically.
- To change `main_wad_file`, the player uses `RESET SETUP` and confirms setup again.

Reason:
- Setup reuse identity should not drift when the player only adjusts helper WAD order.

### Setup Reuse Rule
- Reuse existing multi-WAD setup only when:
  - `main_wad_hash` matches
  - every filename in saved `wad_order` still exists in the resolved content root
  - there is no duplicate-name ambiguity
- If extra `.wad` files are present but the saved launch set is still valid:
  - keep the existing setup
  - optionally show a non-blocking `NEW FILES DETECTED` style notice in detail UI
- Invalidate the saved setup when:
  - the main PWAD changes
  - any saved `wad_order` member is missing
  - duplicate-name ambiguity appears
  - the folder becomes `[UNSUPPORTED]` or otherwise non-launchable
- Renaming the mod folder does not reuse setup. It is treated as a new entry.

## Discovery UX

### Scanning Feedback
- If discovery, hashing, or classification takes longer than a short frame budget, show a visible waiting state.
- Minimum v1 message:
  - `SCANNING MODS...`
- The menu must not look blank or frozen while WAD hashes are being computed.

Reason:
- Initial hash computation for large WADs can take noticeable time.
- Users should understand that the game is working, not hung.

## Operational Observability

### Required Log Points
- Write concise diagnostic lines to `Player.log` for these events:
  - mod discovery start and finish
  - per-entry resolved state
  - setup confirm
  - `CONTINUE` unavailable due to save mismatch
  - startup-time config deletion

### Minimum Fields
- Each diagnostic line should include, when available:
  - `entry_id`
  - resolved `state`
  - mismatch reason or GC action

Reason:
- Support after release depends on being able to reconstruct why an entry became `[SETUP]`, `[UNSUPPORTED]`, `[LAYOUT]`, or why `CONTINUE` disappeared.
- The UI alone is not enough for remote diagnosis.

## Runtime Data Model

### Entry Model

```json
{
  "entry_id": "alien_vendetta",
  "display_name": "Alien Vendetta",
  "folder_name": "alien_vendetta",
  "content_root": "wad/mods/alien_vendetta",
  "wad_files": ["AV.WAD", "AVMOVFIX.WAD"],
  "required_iwad_family": "doom2",
  "manifest_hash": "sha1(...)",
  "state": "ready-multi"
}
```

### Entry Identity
- UI/runtime identity is `entry_id`
- `entry_id` is the direct child folder name under `wad/mods`
- `display_name` comes from:
  - internal config `display_name` if present
  - otherwise `entry_id`
- internal config attachment comes from `entry_id`

### Derived Fields
- `required_iwad_family` is a derived field of the current launch set.
- It is not authoritative persistent user data.
- UI compatibility checks must read the current derived value from `DoomModEntry`.

### Runtime Loadout
- Replace `enabledModFiles` with `selectedModId`

```json
{
  "selectedIwadFile": "freedoom2.wad",
  "selectedModId": "alien_vendetta",
  "selectedSkill": 3
}
```

### Loadout Rule
- If `selectedModId` no longer resolves to a discovered entry on load, clear the selected mod.
- Renaming the mod folder is treated as creating a new entry and does not preserve selection.

## Manifest Hash

### Purpose
- Track launch-relevant content changes safely
- Prevent stale rules and stale saves from silently reusing old assumptions

### Hash Input
- canonical ordered tuple list of:
  - relative `.wad` filename in launch set
  - file size
  - last write utc
  - content sha1
- plus launch-relevant internal config fields only:
  - ordered `wad_order`

### Explicit Exclusion
- Do not include cosmetic-only fields such as `display_name` in `manifest_hash`.

Reason:
- Changing the visible label must not invalidate rules or save identity.

### Hash Reuse / Performance
- Full content SHA1 must not be recalculated for every file on every `CONFIGURE MODS` open.
- Cache per-file hash metadata inside `mod_rules.json`:
  - relative filename
  - file size
  - last write utc
  - cached content sha1
- Reuse cached content sha1 when size and last write utc are unchanged.
- Recalculate content sha1 only for changed members.

## Dependency Classification Model

### Values
- `doom1`
- `doom2`
- `any`
- `unknown`

### Classification Unit
- Classification is performed per entry launch set, not per raw folder and not per individual file.
- The output of classification becomes the base `required_iwad_family` of that entry.
- `required_iwad_family` must always be treated as derived from the launch set, never as an independently edited setting.

### Input Set
- For `ready-multi`, classify only the `.wad` files listed in internal `wad_order`
- For `ready-single`, classify the single `.wad`

### Inference Logic
- Aggregate map lump evidence across the launch set:
  - only `MAPxx` -> `doom2`
  - only `E#M#` -> `doom1`
  - both styles -> `any`
  - no map lumps or parse failure -> `unknown`

### Source Of Truth
- The only authoritative source for dependency compatibility is the current launch set:
  - single-WAD: that WAD
  - multi-WAD: the saved `wad_order` launch set
- `required_iwad_family` is derived from that launch set.
- Any cached classification may be discarded and recomputed without losing correctness.

### Effective Family Rule
- UI compatibility checks use `effective_required_iwad_family`.
- Compute it as:
  1. classify the current launch set
  2. if a manual override exists for the current `manifest_hash`, replace the classified value with that override
  3. otherwise use the classified value

Reason:
- Manual override must have one clear priority rule.
- `[---]`, `[?]`, and launch validation must all use the same final family value.

## Rule Storage

### File
- `BepInEx/config/chitsii.elin_justdoomit/profiles/mod_rules.json`

### Role
- `mod_rules.json` is a classification cache, not a source of truth for mod setup.
- It may also store user manual override for a specific `manifest_hash`.

### Identity Model
- Cache/override identity is `manifest_hash`, not `entry_id`
- `entry_id` is only metadata for current UI mapping

### Rule Semantics
- `manual` rules are tied to the exact `manifest_hash`
- if launch-relevant content changes, the old manual rule no longer applies automatically
- if the same launch set appears again under another folder name, the same rule may be reused safely
- cache miss or deleted cache entry must trigger reclassification from the current launch set
- records whose `manifest_hash` is no longer present in any discovered ready entry are garbage-collected on game startup without affecting correctness
- manual override, when present, replaces the classified family for that exact `manifest_hash`

## Internal Config GC

### File Set
- `mod_entry_configs/*.json`

### GC Rule
- On game startup, immediately delete config files whose `entry_id` is not referenced by any currently discovered multi-WAD candidate.
- If a config file is malformed, ignore it for matching and delete it during the same GC pass.
- If runtime loadout points to a deleted or missing `selectedModId`, clear `selectedModId` during loadout reconciliation.

Reason:
- Internal config should not accumulate forever after folder deletion or restructuring.
- The rule should stay simple and predictable.
- Startup-only cleanup is easier to explain than delayed stale state.

## Save Identity

### Save Slot Key
- Save slot identity must include:
  - selected IWAD file
  - selected entry `manifest_hash`
- `CONTINUE` must be offered only when the exact save slot key matches the current menu selection.
- `manifest_hash` alone is not sufficient for save reuse.

Reason:
- The loaded DOOM save already carries its gameplay state.
- Changing the menu skill after a save exists must not hide `CONTINUE` for the same IWAD/mod pair.

### Continue UX
- If any persistent save exists for the current arcade profile but does not match the current IWAD/mod selection, show a clear reason in the detail area.
- Minimum v1 detail examples:
  - `Saved run exists for: Alien Vendetta / Freedoom Phase 2`
  - `Change GAME or MOD to continue that save`
- The user must understand that the save still exists and why `CONTINUE` is currently unavailable.

## User Flow

1. User opens `OPEN MOD FOLDER`
2. User reads the included guide text
3. User creates one folder per external mod under `wad/mods`
4. User places the mod files inside that folder
5. User opens `CHANGE GAME` and selects the correct IWAD family first
6. User opens `CONFIGURE MODS`
7. If a folder has one `.wad`, it is immediately usable
8. If a folder has multiple `.wad` files, the game opens a setup flow and writes internal config automatically after confirmation
9. Ready entries go through dependency classification
10. User selects one active mod entry
11. User starts with `START OVER` or `CONTINUE`
12. If `CONTINUE` is unavailable, the menu explains which saved run exists and what selection is needed to resume it

## UI Design

### Single Source Of Truth
- UI state ids must be defined once in code and reused by:
  - menu rendering
  - detail-area explanations
  - any other in-game screen that references the same state
- Prefer one shared constant/source for state ids inside runtime code.

Reason:
- Operationally, state renames such as `[SIDECAR]` -> `[UNSUPPORTED]` must not drift across screens and languages.

### Main Screen
- `MODS` line shows selected entry display name

### Configure Mods Screen States
- `[ON ]` selected and valid
- `[OFF]` valid and not selected
- `[---]` valid but incompatible with selected IWAD family
- `[?]` valid but dependency unknown
- `[SETUP]` multi-WAD folder needs setup
- `[ERR ]` internal config invalid
- `[UNSUPPORTED]` folder contains unsupported non-WAD launch files
- `[LAYOUT]` folder detected but layout not recognized

### Per-Entry Actions
- `ready-multi` entries may expose an additional per-entry action:
  - `RESET SETUP`
- `RESET SETUP` belongs to the currently focused `ready-multi` entry only.
- It is not a global command and not a normal row toggle.

### Selection Rules
- `[ON]`, `[OFF]`, `[?]` are selectable
- `[SETUP]` opens setup flow
- focused `ready-multi` entry may open its per-entry action region
- `RESET SETUP` requires a separate explicit selection step plus confirmation
- `[---]`, `[ERR]`, `[UNSUPPORTED]`, `[LAYOUT]` are not directly selectable for launch

### Detail Area
- When a row is selected, show:
  - display name
  - files that will be launched, if any
  - setup/error reason if not playable
- when `RESET SETUP` is focused, show that it will forget saved file order and main PWAD choice
- When `CONTINUE` is unavailable due to selection mismatch, show the saved-run mismatch reason in the main detail area.

## Error Handling

### Empty Folder / No-WAD Folder
- Hidden from UI

### Multiple WADs Without Config
- Show as `[SETUP]`

### Invalid Internal Config
- Show as `[ERR ]`

### Unsupported Non-WAD Inputs
- If the resolved content root contains direct `.deh`, `.bex`, `.pk3`, or `.pk7` files, show as `[UNSUPPORTED]`
- Example reason:
  - `This mod includes non-WAD files that v1 does not support`
- Detail area should also tell the player what to do next:
  - `This mod cannot be launched in this version`
  - `Use a WAD-only mod, or remove the unsupported files from this folder`

### Unrecognized Layout
- Show as `[LAYOUT]`
- Example reason:
  - `WAD files were found only in deeper subfolders`
  - `Folder structure is not supported`

### Oversized Member
- Size limit remains per `.wad` file at 128 MB
- Any failing member pushes classification to `unknown`
- If a listed member cannot be resolved for launch, the entry becomes `[ERR ]`

### Timeout
- Timeout remains per file at 2 seconds
- Timeout during classification yields `unknown`

## Compatibility Decision
- Clean design takes priority over backward compatibility
- New folder-based workflow is the only supported user model
- Loose playable `.wad` files directly under `wad/mods` are no longer selectable mods
- Existing `enabledModFiles` selections may be discarded on first load of the new format
- v1 support target is WAD-only PWAD workflows
- `.deh` / `.bex` / `.pk3` / `.pk7` inputs are out of scope for v1 and must be called out clearly in UI/docs

## Implementation Outline

1. Introduce a new `DoomModEntry` model with `state`, `content_root`, `display_name`, `wad_files`, and `manifest_hash`
2. Replace flat `FindPwads()` with entry discovery under `wad/mods/*/`
3. Add one-level wrapper-folder resolution plus `error-layout` reporting
4. Add internal entry-config read/write under `BepInEx/config/.../mod_entry_configs/`
5. Add one-click multi-WAD setup flow with simple reorder/include controls
6. Replace runtime loadout field `enabledModFiles` with `selectedModId`
7. Rewrite launch resolution to expand one selected entry into multiple `-file` arguments
8. Rewrite `mod_rules.json` to store rules by `manifest_hash` plus per-file hash cache
9. Update save slot key generation to include `manifest_hash`
10. Update `CONFIGURE MODS` UI to list entry folders and setup/error/layout states
11. Treat folder rename as a new entry and clear stale `selectedModId` on load
12. Add startup-time immediate `mod_entry_configs` garbage collection
13. Add `RESET SETUP` action for configured multi-WAD entries
14. Add confirmation dialog for `RESET SETUP`
15. Add visible scanning feedback during expensive discovery/hash work
16. Add `CONTINUE` mismatch reason display
17. Update guide text so users only create folders and copy files

## Acceptance Criteria
- A folder containing exactly one `.wad` appears as one selectable mod without extra setup
- A folder containing multiple `.wad` files appears as `[SETUP]` instead of asking the user to edit JSON
- Accepting setup writes internal config under `BepInEx/config/...`, not under `wad`
- Users can adjust order and inclusion in the setup UI without touching files
- Configured multi-WAD entries expose `RESET SETUP` in-game and do not require file deletion to redo setup
- `RESET SETUP` is visually separated from ordinary mod selection/toggle rows
- `RESET SETUP` always asks for confirmation before deleting saved setup
- `display_name` changes do not invalidate save identity or rule identity
- Renaming a mod folder is treated as a different entry, but setup reset itself is an in-game action
- Reordering helper WADs does not change `main_wad_file` implicitly
- After setup confirmation, the entry becomes selectable immediately in the same `CONFIGURE MODS` session
- Unsupported non-WAD inputs are visible as `[UNSUPPORTED]` instead of silently disappearing
- Guides and UI clearly state that v1 supports WAD-only workflows, not `.deh` / `.bex` / `.pk3` / `.pk7`
- Helper WADs never appear as independent rows
- Save identity changes when launch-relevant content changes, even if folder/file names stay the same
- `CONTINUE` availability does not depend on the current menu skill
- Long discovery/hash work shows `SCANNING MODS...` instead of looking frozen
- If a save exists but does not match current IWAD/mod selection, the menu explains what to switch to
- Temporary folder mistakes do not immediately erase multi-WAD setup
- `Player.log` contains enough data to explain discovery state, setup confirmation, startup GC, and `CONTINUE` mismatch
- QA can verify state names from one canonical source instead of per-screen hardcoding
- Clearing `mod_rules.json` does not lose setup and only forces dependency reclassification from the current launch set
- UI compatibility state is driven by current derived `required_iwad_family`, not by stale persisted dependency data
- Manual override changes the final family seen by UI compatibility checks
- QA case: a large WAD shows `SCANNING MODS...` before the list becomes interactive
- QA case: changing only helper WAD timestamps does not discard setup when the main PWAD is unchanged
- QA case: `RESET SETUP` returns a configured multi-WAD entry to `[SETUP]` immediately
- QA case: `RESET SETUP` clears selection when that entry was selected
- QA case: `RESET SETUP` cannot be triggered from the normal mod-toggle hit area by mistake
- QA case: canceling the `RESET SETUP` confirmation leaves config, selection, and row state unchanged
- QA case: after manual override, `[---]` and `[?]` reflect the override rather than the raw classified value
- QA case: `[UNSUPPORTED]` entries cannot be launched and explain what to do next
